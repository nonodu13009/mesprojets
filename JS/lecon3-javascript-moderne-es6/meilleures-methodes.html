Les Meilleures Méthodes pour Mieux Coder

Leçon 3 · Chapitre 29 · Écrire du JavaScript propre et robuste

Pourquoi ce chapitre ?

Tu connais maintenant la syntaxe moderne. Mais entre "du code qui marche" et "du code de qualité", il y a un ensemble d'habitudes que tous les bons développeurs partagent. Ce chapitre les rassemble : c'est ta checklist qualité, à relire régulièrement.

1. const par défaut, let si nécessaire

N'utilise jamais var. Déclare tout en const, et passe à let seulement si la variable doit être réassignée : 

// ✅ const : la valeur ne changera pas
const TAUX_TVA = 0.2;
const utilisateur = { nom: "Nogaro" };

// ✅ let : le compteur va changer
let compteur = 0;
compteur++;

// ❌ var : portée confuse, hissage surprenant — à bannir
var ancien = "non merci";

Bonus : const documente ton intention. Quand tu relis ton code, tu sais immédiatement quelles valeurs peuvent bouger.

2. Des noms explicites

Le code est lu bien plus souvent qu'il n'est écrit. Un bon nom évite un commentaire : 

// ❌ Illisible
const d = 30;
const fn = (a, b) => a * b;

// ✅ Le code se lit comme une phrase
const dureeEnJours = 30;
const calculerSurface = (largeur, hauteur) => largeur * hauteur;
  • Variables et fonctions en camelCase : prixTotal, calculerTTC
  • Booléens qui posent une question : estActif, aUnCompte
  • Fonctions qui commencent par un verbe : obtenirUtilisateur, afficherPanier
  • Constantes "magiques" en MAJUSCULES : TAUX_TVA, MAX_TENTATIVES

3. Égalité stricte : === toujours

Le double égal == convertit les types avant de comparer, ce qui produit des résultats surprenants. Le triple égal === compare valeur et type : 

console.log(5 == "5");   // true 😱 (conversion implicite)
console.log(5 === "5");  // false ✅ (types différents)

console.log(0 == false);  // true 😱
console.log(0 === false); // false ✅

4. Template literals plutôt que concaténation

Les backticks ` rendent les chaînes composées bien plus lisibles que les + en cascade : 

const prenom = "Jean-Michel";
const age = 30;

// ❌ Concaténation : facile de se tromper dans les espaces
const phrase = "Je suis " + prenom + " et j'ai " + age + " ans.";

// ✅ Template literal : lisible d'un coup d'œil
const phrase2 = `Je suis ${prenom} et j'ai ${age} ans.`;

5. Des fonctions courtes qui font UNE chose

Une fonction de 40 lignes qui valide, calcule et affiche est impossible à tester et à réutiliser. Découpe : 

// ✅ Trois petites fonctions, chacune avec une responsabilité
const validerPrix = prix => prix > 0;

const calculerTTC = prixHT => prixHT * 1.2;

const formaterPrix = prix => `${prix.toFixed(2)} €`;

// Et on les compose
const prixHT = 49;
if (validerPrix(prixHT)) {
    console.log(formaterPrix(calculerTTC(prixHT))); // "58.80 €"
}

6. Sortir tôt (early return)

Plutôt que d'imbriquer des if dans des if, traite les cas d'erreur en premier et sors immédiatement : 

// ❌ Imbrication profonde : dur à suivre
const traiterCommande = (commande) => {
    if (commande) {
        if (commande.articles.length > 0) {
            if (commande.paiementValide) {
                return "Commande expédiée";
            }
        }
    }
    return "Erreur";
};

// ✅ Early return : chaque cas d'erreur est traité puis oublié
const traiterCommande2 = (commande) => {
    if (!commande) return "Commande manquante";
    if (commande.articles.length === 0) return "Panier vide";
    if (!commande.paiementValide) return "Paiement refusé";

    return "Commande expédiée";
};

7. Ne pas se répéter (DRY)

Don't Repeat Yourself : si tu copies-colles un bloc de code une troisième fois, transforme-le en fonction. Un bug corrigé à un seul endroit vaut mieux qu'un bug à chasser dans cinq copies. 

// ❌ La même logique copiée-collée
console.log(`Clavier : ${(49 * 1.2).toFixed(2)} €`);
console.log(`Souris : ${(29 * 1.2).toFixed(2)} €`);

// ✅ Une fonction réutilisable
const afficherPrixTTC = (nom, prixHT) =>
    console.log(`${nom} : ${(prixHT * 1.2).toFixed(2)} €`);

afficherPrixTTC("Clavier", 49);
afficherPrixTTC("Souris", 29);

8. Préférer les méthodes de tableaux aux boucles

Quand c'est possible, find / filter / map (chapitre précédent) expriment ton intention mieux qu'une boucle for : le nom de la méthode dit ce que fait le code. 

const notes = [12, 8, 15, 17, 9];

// ❌ Boucle : il faut lire tout le corps pour comprendre
const bonnesNotes = [];
for (let i = 0; i < notes.length; i++) {
    if (notes[i] >= 10) {
        bonnesNotes.push(notes[i]);
    }
}

// ✅ filter : l'intention est dans le nom
const bonnesNotes2 = notes.filter(note => note >= 10);

Ta checklist qualité

  • const par défaut, let si besoin, jamais var
  • Noms explicites : le code se lit comme une phrase
  • Toujours ===, jamais ==
  • Template literals `${ }` pour composer des chaînes
  • Fonctions courtes, une seule responsabilité
  • Early return plutôt que des if imbriqués
  • DRY : factoriser dès la troisième répétition
  • find / filter / map plutôt que des boucles quand c'est naturel

À toi de jouer

  1. Reprends un de tes anciens scripts (leçon 1 ou 2)
  2. Remplace les var et les let inutiles par const
  3. Renomme les variables trop courtes (d, x, tmp...)
  4. Remplace les concaténations + par des template literals
  5. Cherche un if imbriqué et réécris-le en early return
  6. Repère une boucle for remplaçable par filter ou map
