Les Fonctions de Rappel (Callbacks)

Leçon 5 · Chapitre 33 · Le point de départ de l'asynchrone

C'est quoi, un callback ?

Un callback (fonction de rappel) est une fonction passée en argument à une autre fonction, pour être exécutée plus tard. Tu en utilises déjà depuis la leçon 3 sans le savoir : 

// La fléchée passée à forEach est... un callback !
[1, 2, 3].forEach(n => console.log(n));

// Pareil pour map, filter, find...
const doubles = [1, 2, 3].map(n => n * 2);

L'idée clé : tu ne l'appelles pas toi-même (pas de parenthèses !), tu la confies à une autre fonction qui l'appellera au bon moment.

Le temps entre en jeu : setTimeout

setTimeout exécute un callback après un délai (en millisecondes). C'est l'exemple le plus simple d'asynchrone : 

console.log("1. Je commence");

setTimeout(() => {
    console.log("3. Deux secondes plus tard...");
}, 2000);

console.log("2. Je continue sans attendre !");

// Ordre d'affichage : 1, 2, puis 3

Regarde bien l'ordre : JavaScript n'attend pas les 2 secondes. Il note le rendez-vous, continue son travail, et exécute le callback quand le minuteur sonne. C'est ça, l'asynchrone.

Piège classique : écrire setTimeout(maFonction(), 2000) avec des parenthèses exécute la fonction immédiatement. On passe la fonction elle-même : setTimeout(maFonction, 2000).

Les callbacks d'événements

Le DOM (leçon 4) utilise aussi les callbacks : c'est comme ça qu'on réagit aux actions de l'utilisateur, avec addEventListener : 

const titre = document.querySelector("h1");

// "Quand on clique sur le titre, exécute ce callback"
titre.addEventListener("click", () => {
    titre.textContent = "Tu m'as cliqué !";
});

Même logique que setTimeout : on ne sait pas quand le clic arrivera, alors on confie un callback qui sera appelé à ce moment-là. Teste ce code dans la console, ici même !

Simuler une opération lente

Écrivons notre propre fonction asynchrone : elle simule un appel serveur qui met 1,5 seconde à répondre, puis "rappelle" avec le résultat : 

const chargerUtilisateur = (id, callback) => {
    console.log(`Chargement de l'utilisateur ${id}...`);
    setTimeout(() => {
        const utilisateur = { id, nom: "Nogaro", prenom: "Jean-Michel" };
        callback(utilisateur); // on "rappelle" avec le résultat
    }, 1500);
};

chargerUtilisateur(1, (utilisateur) => {
    console.log(`Reçu : ${utilisateur.prenom} ${utilisateur.nom}`);
});

C'est exactement le contrat d'un callback asynchrone : "lance l'opération, et quand c'est prêt, appelle cette fonction avec le résultat".

La limite : le callback hell

Le problème surgit quand les opérations s'enchaînent : charger un utilisateur, PUIS ses commandes, PUIS le détail d'une commande... Chaque étape s'imbrique dans la précédente : 

// La "pyramide de la mort" 😱
chargerUtilisateur(1, (utilisateur) => {
    chargerCommandes(utilisateur.id, (commandes) => {
        chargerDetail(commandes[0].id, (detail) => {
            chargerFacture(detail.id, (facture) => {
                console.log(facture);
                // ... et la gestion d'erreurs de chaque niveau ?
            });
        });
    });
});

Ce code en escalier est surnommé le callback hell : difficile à lire, difficile à modifier, et cauchemardesque pour gérer les erreurs. C'est précisément pour résoudre ça qu'ES6 a introduit les promesses — le sujet du prochain chapitre.

À toi de jouer

  1. Ouvre la console (F12 → Console)
  2. Affiche un message après 3 secondes avec setTimeout
  3. Écris trois console.log autour d'un setTimeout et prédis l'ordre d'affichage avant d'exécuter
  4. Ajoute un addEventListener("click", ...) sur le h1 de cette page
  5. Recrée chargerUtilisateur et appelle-la avec ton propre callback
  6. Imbrique deux setTimeout l'un dans l'autre pour ressentir le début du callback hell

La suite

Tu as compris le mécanisme fondamental : confier une fonction qui sera rappelée plus tard. Et tu as vu sa limite : l'imbrication. Les promesses gardent l'idée mais aplatissent la pyramide.
