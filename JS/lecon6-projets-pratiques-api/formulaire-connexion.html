Formulaire de Connexion et Inscription Animé

Leçon 6 · Projet 42 · Validation, animations et bascule

Le brief

Construire le composant le plus universel du web : un double formulaire connexion / inscription avec une bascule animée entre les deux, une validation en temps réel des champs, et des messages d'erreur élégants. Pas de serveur ici : tout se joue côté interface — le back-end viendra plus tard dans ton parcours.

Ce que tu vas pratiquer

  • Événements de formulaire : submit, input, blur (leçon 4)
  • Validation : expressions conditionnelles, longueur, format email (leçon 2)
  • classList.toggle pour la bascule animée entre les deux panneaux (leçon 4)
  • Transitions et animations CSS déclenchées par JavaScript

Fonctionnalités attendues

  • Deux formulaires : connexion (email, mot de passe) et inscription (nom, email, mot de passe, confirmation)
  • Bascule animée entre les deux (glissement ou fondu) via un lien "Pas encore de compte ?"
  • Validation en temps réel : bordure rouge + message sous le champ invalide, vert quand c'est bon
  • Règles : email au bon format, mot de passe ≥ 8 caractères, confirmation identique
  • Bouton "voir le mot de passe" (œil qui bascule le type de l'input)
  • Soumission bloquée tant que tout n'est pas valide

Le cœur du projet : la validation

Le réflexe pro : une fonction de validation par règle, et une fonction générique qui affiche ou efface l'erreur d'un champ : 

// Une règle = une petite fonction pure qui renvoie un message ou null
const validerEmail = (valeur) => {
    if (valeur.trim() === "") return "L'email est requis";
    if (!valeur.includes("@") || !valeur.includes(".")) return "Format d'email invalide";
    return null; // null = tout va bien
};

const validerMotDePasse = (valeur) => {
    if (valeur.length < 8) return "8 caractères minimum";
    return null;
};

Note l'early return (leçon 3, chapitre 29) : chaque règle est testée puis oubliée, pas d'imbrication.

Plan de construction

  1. HTML : les deux <form> dans un conteneur, chaque champ accompagné d'un <small> vide pour le message d'erreur
  2. CSS : la carte, les états .champ-erreur / .champ-valide (bordures), et l'animation de bascule entre les panneaux
  3. Bascule : les liens "Créer un compte" / "Déjà inscrit ?" font un classList.toggle sur le conteneur — le CSS fait le reste
  4. Fonctions de validation : une par règle, testées d'abord en console
  5. Validation en direct : sur l'événement blur (le champ perd le focus) de chaque input, applique la règle et affiche le message
  6. Soumission : sur submit, event.preventDefault(), valide TOUS les champs, et affiche un message de succès si tout passe

Les pièges classiques

  • La page se recharge à la soumission : c'est le comportement par défaut d'un <form> — event.preventDefault() est indispensable
  • Valider seulement à la soumission : l'utilisateur préfère savoir tout de suite — d'où le blur champ par champ
  • Confirmation de mot de passe : cette règle dépend de DEUX champs — pense à la revalider quand l'un OU l'autre change
  • Messages qui s'accumulent : écrase toujours le contenu du <small>, ne fais pas d'ajout

Pour aller plus loin

  • Jauge de force du mot de passe (réutilise la logique du projet 38 !)
  • Validation email plus stricte avec une expression régulière
  • Animation "secousse" (shake) du formulaire quand la soumission échoue
  • Simule une connexion asynchrone : bouton en état "Connexion..." pendant 1,5 s avec setTimeout + promesse
