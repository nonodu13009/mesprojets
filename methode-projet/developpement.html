Développement

Étape 4 · Coder avec Claude Code — du setup au premier écran

Le moment de coder

Tu as un brief, un cadrage, des maquettes. Tu sais exactement ce que tu construis. C'est maintenant que Claude Code entre en jeu. Ton rôle : piloter. Le rôle de Claude Code : exécuter.

Cette étape suit une séquence précise. Ne brûle pas les étapes — un projet bien structuré au départ économise des heures de galère ensuite.

Phase 1 : Setup du projet

Avant d'écrire la moindre fonctionnalité, pose les fondations. Demande à Claude Code : 

Crée un projet Next.js avec :
- TypeScript
- Tailwind CSS
- shadcn/ui
- Structure App Router (app/)
- Un layout de base avec header et sidebar

Initialise aussi un repo git local.

Claude Code va créer toute l'arborescence, installer les dépendances, configurer les fichiers. Tu n'as qu'à valider.

Première chose à faire ensuite : crée un fichier CLAUDE.md à la racine avec ton brief, tes personas, ta stack et ton système de design. Claude Code le lira automatiquement à chaque session.

Phase 2 : Le CLAUDE.md

C'est le fichier le plus important de ton projet. Il donne à Claude Code tout le contexte pour travailler efficacement. Structure recommandée : 

# CLAUDE.md — [Nom du projet]

## 1. Brief
[Colle ton brief ici — problème, persona, solution]

## 2. Stack
- Next.js 15 (App Router)
- TypeScript strict
- Tailwind CSS + shadcn/ui
- Supabase (auth + BDD)
- Vercel (déploiement)

## 3. Design System
- Couleur principale : #0071e3
- Fond : blanc #fff, surfaces #f5f5f7
- Typo : system font (-apple-system)
- Border radius : 12px cards, 980px boutons
- Style : Apple HIG, minimaliste

## 4. Conventions
- const par défaut, let si réassignation, jamais var
- Composants en PascalCase, fichiers en kebab-case
- Un composant par fichier
- Pas de console.log en production

## 5. Structure
app/
├── layout.tsx        (layout global)
├── page.tsx          (dashboard)
├── leads/
│   ├── page.tsx      (liste des leads)
│   └── [id]/page.tsx (détail d'un lead)
components/
├── ui/               (shadcn/ui)
├── lead-card.tsx
├── lead-table.tsx
└── status-badge.tsx

Phase 3 : Développer par user story

Ne demande jamais « code-moi toute l'app ». Procède user story par user story. C'est plus fiable, plus facile à vérifier, et plus facile à corriger. 

// Session 1 : le layout
"Crée le layout global avec un header qui affiche le nom
de l'app et un bouton 'Ajouter un lead'. Pas de sidebar
pour le MVP. Style Apple HIG."

// Session 2 : la liste des leads
"Implémente cette user story : En tant qu'agent, je veux
voir tous mes leads sur une page, avec colonnes Nom, Email,
Téléphone, Statut et Date. Utilise un tableau shadcn/ui.
Données mockées en dur pour l'instant."

// Session 3 : les filtres
"Ajoute des filtres par statut au-dessus du tableau.
4 boutons : Tous, Nouveaux, En cours, Gagnés.
Le filtre actif est en bleu."

// Session 4 : la BDD
"Remplace les données mockées par Supabase.
Crée une table 'leads' avec les colonnes nécessaires.
Connecte le tableau à la BDD."

Règle : chaque session = 1 fonctionnalité. Teste et valide avant de passer à la suivante. Commite après chaque session réussie (/commit).

La boucle de travail

Voici le cycle que tu répètes pour chaque fonctionnalité :

1. Demanderdécris la user story 2. Vérifierteste dans le navigateur 3. Ajustercorrige / affine 4. Commitersauvegarde le code
Étape Toi Claude Code
Demander Décris ce que tu veux en langage naturel Écrit le code
Vérifier Ouvre l'app dans Chrome, teste —
Ajuster « Le bouton est trop petit », « Ajoute un hover » Modifie le code
Commiter Tape /commit Stage, commit, push

Commandes Claude Code essentielles

Commande Ce qu'elle fait Quand l'utiliser
/commit Stage + commit + push avec message auto Après chaque fonctionnalité validée
/help Affiche l'aide Quand tu es perdu
⌘ + R Efface le contexte de conversation Quand Claude Code perd le fil

Erreurs courantes

Erreur Conséquence Solution
Demander trop en une fois Code fragile, difficile à débugger 1 user story = 1 session
Ne pas tester entre chaque étape Les bugs s'accumulent Ouvre le navigateur après chaque changement
Oublier de commiter Tu perds du travail /commit après chaque validation
Pas de CLAUDE.md Claude Code n'a pas le contexte du projet Crée-le en premier, mets-le à jour

La suite

Ton app fonctionne en local. Les fonctionnalités du MVP sont en place. Avant de la mettre en ligne, il faut la tester sérieusement : est-ce que tout marche sur mobile ? Que se passe-t-il avec des données vides ? Les erreurs sont-elles gérées proprement ?
