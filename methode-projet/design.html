Design

Étape 3 · Donner forme à l'app avant d'écrire du code

Pourquoi designer avant de coder ?

La tentation est forte de sauter directement dans le code. Mais modifier un wireframe prend 2 minutes. Modifier du code prend 2 heures. Le design te permet de tester des idées gratuitement, de voir les problèmes d'UX avant qu'ils ne deviennent des bugs, et de donner une direction visuelle claire à Claude Code.

Tu n'as pas besoin d'être designer. Tu as Claude Design.

Les 3 niveaux de design

Ne passe pas directement à la maquette finale. Il y a 3 niveaux, chacun plus détaillé que le précédent :

Wireframestructure (squelette) → Maquettevisuel (apparence) → Prototypeinteractif (cliquable)
Niveau Ce que c'est Outil Temps
Wireframe Des boîtes grises, pas de couleurs. « Quoi va où » Papier, Claude Design 15-30 min
Maquette Couleurs, typo, vrais textes. « À quoi ça ressemble » Claude Design, Figma 1-2 h
Prototype Maquette cliquable. « Comment ça se navigue » Claude Code (HTML/CSS direct) 2-3 h

Pour un MVP : wireframe + maquette suffisent. Le prototype n'est utile que si tu dois montrer l'app à quelqu'un avant de la coder (investisseur, associé, client).

Étape A : les wireframes (papier)

Prends une feuille A4 et dessine les écrans principaux de ton app. Pas besoin de talent artistique — des rectangles et du texte suffisent. L'objectif est de répondre à ces questions :

  1. Combien d'écrans a mon app ? (liste-les)
  2. Que voit l'utilisateur sur chaque écran ?
  3. Comment navigue-t-il d'un écran à l'autre ?
  4. Où sont les actions principales (boutons, formulaires) ?
// Exemple — LeadTracker : liste des écrans

Écran 1 : Dashboard (liste des leads + filtres + stats rapides)
Écran 2 : Détail d'un lead (infos + historique + actions)
Écran 3 : Formulaire d'ajout/édition de lead
Écran 4 : Login (email + mot de passe)

4 écrans. Pas 20. C'est un MVP.

Étape B : maquettes avec Claude Design

Une fois tes wireframes papier validés, utilise Claude Design (dans claude.ai) pour générer des maquettes visuelles. Le prompt est crucial — plus tu es précis, meilleur est le résultat. 

Crée une maquette pour un dashboard de gestion de leads.

Style : Apple HIG, minimaliste, fond blanc, accent bleu #0071e3.
Typo : SF Pro / system font.

L'écran contient :
- En haut : barre de navigation avec logo + nom de l'app
- En dessous : 4 compteurs (Nouveaux, Contactés, En cours, Gagnés)
- Au centre : tableau des leads avec colonnes :
  Nom | Email | Téléphone | Statut | Date | Actions
- Chaque ligne a un badge de statut coloré
- Bouton "Ajouter un lead" en haut à droite (bleu, pill shape)
- Filtres par statut au-dessus du tableau

Desktop, 1440px de large.
Montre des données réalistes (noms français, vrais emails).

Itère : le premier résultat ne sera pas parfait. Demande des ajustements : « Plus d'espace entre les lignes », « Change le badge rouge en orange », « Ajoute une sidebar ». 3-4 itérations suffisent généralement.

Le système de design

Avant de coder, fige tes choix visuels. Ça évitera les incohérences et donnera des instructions claires à Claude Code. Définis au minimum :

Élément Décision Exemple
Couleur principale L'accent de l'app #0071e3
Couleur fond Background principal #ffffff
Couleur texte Corps de texte #1d1d1f
Police Titres + corps SF Pro / system
Border radius Coins des éléments 12px cards, 980px boutons
Composants UI Bibliothèque de composants shadcn/ui

Ces choix iront dans ton CLAUDE.md. Claude Code les appliquera automatiquement à chaque fichier qu'il génère.

Le livrable

  1. La liste des écrans de ton app
  2. Les maquettes des écrans principaux (screenshots Claude Design)
  3. Le système de design (couleurs, typo, composants)
  4. Le flux de navigation (quel écran mène à quel autre)

La suite

Tu as un brief, un cadrage et des maquettes. Tu sais exactement quoi construire, pour qui, avec quoi, et à quoi ça ressemble. Il est temps d'ouvrir le terminal. L'étape suivante : le développement avec Claude Code — setup du projet, architecture des fichiers et premiers écrans fonctionnels.
