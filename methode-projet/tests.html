Tests

Étape 5 · Vérifier, casser et corriger avant la mise en ligne

Pourquoi tester ?

Ton app fonctionne sur ton écran. Mais fonctionne-t-elle sur un iPhone ? Avec un réseau lent ? Quand l'utilisateur tape n'importe quoi dans un formulaire ? Quand la base de données est vide ?

Tester, ce n'est pas « vérifier que ça marche ». C'est essayer de casser l'app avant que tes utilisateurs ne le fassent.

La checklist de test MVP

Avant chaque mise en ligne, passe systématiquement cette checklist. Tu peux demander à Claude Code de t'aider pour chaque point.

Catégorie Vérification Comment tester
Fonctionnel Chaque user story fonctionne Rejoue chaque scénario manuellement
Cas limites Données vides, textes très longs, caractères spéciaux Tape des valeurs extrêmes dans chaque champ
Mobile L'app est utilisable sur petit écran DevTools → mode responsive (⌘ + ⌥ + I)
Erreurs Les erreurs affichent un message clair Coupe le réseau, soumets des formulaires incomplets
Performance La page charge en moins de 3 secondes DevTools → onglet Lighthouse
Navigation Tous les liens fonctionnent, pas de page 404 Clique sur chaque lien et bouton

Tester avec Claude Code

Tu n'as pas besoin d'écrire les tests toi-même. Demande à Claude Code de le faire. Voici les prompts utiles : 

// Demander un audit rapide
"Passe en revue le code du projet. Identifie les bugs potentiels,
les erreurs non gérées, les problèmes de sécurité et les soucis
de responsive. Liste-les par priorité."

// Demander des tests automatisés
"Écris des tests pour le composant LeadTable.
Teste : affichage avec données, affichage sans données,
filtrage par statut, changement de statut."

// Vérifier le responsive
"Vérifie que toutes les pages sont responsive.
Corrige les problèmes de layout sur mobile (< 640px)."

// Vérifier l'accessibilité
"Ajoute les attributs d'accessibilité manquants :
aria-labels, rôles, alt text, focus visible."

Le test le plus important : le test humain

Demande à quelqu'un (collègue, ami, famille) d'utiliser l'app sans aucune explication. Observe silencieusement :

  1. Est-ce qu'il comprend tout de suite ce que fait l'app ?
  2. Est-ce qu'il trouve les boutons sans aide ?
  3. Est-ce qu'il hésite ou bloque quelque part ?
  4. Quelles questions pose-t-il ?

Chaque hésitation est un bug d'UX. Note-les et corrige-les avant le déploiement.

Le livrable

  1. La checklist cochée (tous les tests passent)
  2. Les bugs corrigés
  3. Les retours du test humain intégrés
  4. Un commit propre : « App prête pour déploiement »

La suite

Ton app est testée, stable, fonctionnelle. Il est temps de la rendre accessible au monde : déploiement. Domaine, hébergement, variables d'environnement — on met tout en ligne avec Vercel.
