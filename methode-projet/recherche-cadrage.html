Cadrage

Étape 2 · Définir les utilisateurs, les fonctionnalités et l'architecture

Du brief au plan

Ton brief dit quoi et pour qui. Le cadrage dit comment. C'est l'étape où tu transformes une idée validée en un plan actionnable — sans encore écrire de code ni dessiner de maquette.

Le cadrage produit 3 livrables : les personas, la liste de fonctionnalités priorisées et le choix de stack technique.

1. Les personas

Un persona est un profil fictif qui représente un type d'utilisateur. Il t'oblige à penser en termes de besoins réels, pas de fonctionnalités abstraites. Limite-toi à 2-3 personas max. 

// Persona 1 — L'agent principal
Nom :       Jean-Michel, 45 ans
Rôle :       Agent général d'assurance
Contexte :   Gère 2 agences, reçoit 15-20 leads/semaine par email
Frustration : Perd des leads, oublie les relances, pas de vue d'ensemble
Besoin :     Voir tous les leads en un coup d'œil, être alerté pour relancer
Tech :       Mac, Chrome, pas dev, utilise Gmail et Slack

// Persona 2 — La collaboratrice
Nom :       Sophie, 30 ans
Rôle :       Chargée de clientèle
Contexte :   Gère le suivi quotidien des prospects
Frustration : Ne sait pas quels leads sont prioritaires
Besoin :     Liste claire de ses tâches du jour, historique des échanges

Prompt Claude : « Voici mon brief [colle-le]. Propose-moi 2-3 personas avec nom, rôle, contexte, frustration, besoin et niveau tech. »

2. Les fonctionnalités — MVP vs V2

C'est ici que tu dois être impitoyable. Tu as certainement 50 idées de fonctionnalités. Tu n'en gardes que 5 pour le MVP (Minimum Viable Product). Le reste va dans la colonne « Plus tard ».

La méthode MoSCoW est simple et efficace :

Priorité Signification Règle
Must Indispensable — l'app ne sert à rien sans 3-5 fonctionnalités max
Should Important — mais on peut lancer sans La V1.1
Could Sympa — si on a le temps La V2
Won't Pas maintenant — on y reviendra peut-être Le backlog 
// Exemple — LeadTracker Assurance

MUST (MVP)
- Afficher la liste des leads avec statut
- Changer le statut d'un lead (nouveau → contacté → gagné/perdu)
- Ajouter un lead manuellement
- Filtrer par statut

SHOULD (V1.1)
- Import automatique depuis Gmail
- Rappels de relance (notifications)
- Historique des actions sur un lead

COULD (V2)
- Dashboard avec statistiques (taux de conversion, etc.)
- Export CSV
- Multi-utilisateurs avec rôles

WON'T (pas maintenant)
- App mobile native
- Intégration CRM Allianz
- Chatbot prospect

Règle d'or : si tu ne peux pas construire ton MVP en une semaine avec Claude Code, c'est que ton scope est trop large. Coupe encore.

3. User stories

Une user story décrit une fonctionnalité du point de vue de l'utilisateur. Le format est toujours le même : 

En tant que [persona],
je veux [action],
afin de [bénéfice].

Exemples :

// User stories MVP
En tant qu'agent, je veux voir tous mes leads sur une page,
afin de savoir combien de prospects j'ai en cours.

En tant qu'agent, je veux changer le statut d'un lead d'un clic,
afin de suivre l'avancement sans effort.

En tant que collaboratrice, je veux filtrer les leads par statut,
afin de me concentrer sur ceux à relancer aujourd'hui.

Ces stories serviront directement de instructions pour Claude Code quand tu développeras. Tu pourras dire : « Implémente cette user story » et l'IA aura tout le contexte.

4. Choix de stack technique

Tu n'as pas besoin de tout comprendre. Tu dois juste savoir quel type de projet tu construis, pour que Claude Code parte sur les bons rails.

Type de projet Stack recommandé Quand l'utiliser
Landing page / site vitrine HTML + CSS + Vercel Pas de compte utilisateur, contenu statique
Web app simple Next.js + Tailwind + Vercel Dashboard, outil interne, CRUD
Web app avec auth + BDD Next.js + Supabase + Vercel Comptes utilisateurs, données persistantes
SaaS complet Next.js + Supabase + Stripe + Vercel Abonnements, paiement, multi-tenants

Conseil : en cas de doute, pars sur Next.js + Tailwind + Supabase + Vercel. C'est le stack le mieux supporté par Claude Code, le plus documenté, et le plus rapide à déployer.

Le livrable

À la fin de cette étape, tu dois avoir :

  1. 2-3 personas avec leurs besoins
  2. Une liste MoSCoW de fonctionnalités
  3. Les user stories du MVP (5-8 max)
  4. Le stack technique choisi

Tout ça rentre dans un seul document — ton cahier de cadrage. Tu le mettras dans le CLAUDE.md de ton projet pour que Claude Code ait toujours le contexte.

La suite

Tu sais maintenant quoi construire, pour qui et avec quoi. Avant d'ouvrir le terminal, il reste une étape cruciale : le design. À quoi ressemblera ton app ? Comment l'utilisateur naviguera-t-il ? C'est le moment d'utiliser Claude Design pour donner forme à ta vision.
